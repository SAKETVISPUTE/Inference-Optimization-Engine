# Custom Autoregressive Generation Engine

This document describes the design, implementation, and verification of our custom autoregressive generation engine.

---

## 1. What was Built
We implemented a custom autoregressive generation loop in PyTorch, bypassing Hugging Face's `model.generate()`. The system supports:
1. **Model & Tokenizer Loading**: Configurable dtype (float32, float16, bfloat16) and device (CPU/CUDA).
2. **Greedy Decoding**: Selecting the highest probability token at each generation step.
3. **Sampling Decoding**: Generating tokens stochastically with:
   * **Temperature**: Scaling logits to control output randomness.
   * **Top-K filtering**: Keeping only the $K$ tokens with the highest probabilities.
   * **Top-P (Nucleus) filtering**: Keeping the minimum set of tokens whose cumulative probability exceeds $P$.
4. **Parity Validation**: Step-by-step verification of generated token IDs against Hugging Face's built-in greedy decoding.

### Key Functions
* `load_model_and_tokenizer(model_id, device, dtype)`: Loads standard causal LM weights and tokenizers.
* `CustomGenerator.generate(...)`: Main generation loop. Tokenizes prompts, manages sequence growth, calls the model forward pass, selects the next token, and checks for termination (EOS token or length limit).
* `CustomGenerator.sample_next_token(logits, temp, top_k, top_p)`: Filters and samples the next token index from the vocabulary logit distribution.

## 3. How it Works (No-Cache Baseline)
At each generation step $t$:
1. The accumulated sequence of tokens $x_{1:t-1}$ is fed into the transformer model.
2. The model performs a full forward pass, computing the representation of all tokens in the sequence from scratch.
3. The logits for the final position (representing the prediction for token $t$) are extracted:
   $$\text{logits}_t = \text{logits}[:, -1, :]$$
4. If temperature is $0.0$, the next token is chosen via greedy decoding:
   $$x_t = \arg\max(\text{logits}_t)$$
5. If temperature $> 0.0$:
   * Logits are scaled: $\text{logits}_t \leftarrow \text{logits}_t / \text{temperature}$.
   * If Top-K $> 0$, logits outside the top-$K$ indices are masked to $-\infty$.
   * If Top-P $> 0.0$, logits are sorted, and those whose cumulative probability sum exceeds $P$ are masked to $-\infty$.
   * A token is sampled from the remaining softmax probability distribution.
6. The token $x_t$ is appended to the input sequence $x_{1:t} = [x_{1:t-1}, x_t]$, and the loop repeats.

> [!NOTE]
> Since we do not have a KV cache in this phase, the attention mechanism recalculates Key and Value representations for all past tokens $x_{1:t-1}$ at every single step $t$. This leads to $O(N^2)$ computations, where $N$ is the total sequence length.

## 4. Validation Results
Validation was performed on CPU using `Qwen/Qwen2.5-0.5B-Instruct` to verify token alignment under greedy decoding. 

* **Test Prompt**: *"State the first three numbers in the sequence of prime numbers."*
* **Baseline Configuration**: Greedy search, max 20 new tokens.
* **Findings**:
  * Hugging Face's `generate` method defaults to the model's `generation_config`, which includes `repetition_penalty = 1.1`. This caused the outputs to diverge at step 15.
  * Disabling the repetition penalty (`repetition_penalty = 1.0` in Hugging Face) resulted in a perfect alignment (Max Logit Difference $< 1.1 \times 10^{-5}$ due to floating-point precision differences).
  * **Tokens generated by both Custom and HF**:
    `[576, 1156, 2326, 10250, 5109, 525, 220, 17, 11, 220, 18, 11, 323, 220, 20, 13, 3555, 374, 279, 1790]`
  * **Decoded Output**: *"The first three prime numbers are 2, 3, and 5. What is the next"*
  * **Result**: **Tokens Match: True**. Parity successfully validated.

## 5. Interview Questions & Answers

### Q1: Why does autoregressive generation become memory-bandwidth bound at batch size 1, and how does this differ from prefill?
During the **Prefill** (pre-computation) phase, the model processes the entire prompt at once. The operations are mostly matrix-matrix multiplications (GEMM), which have high arithmetic intensity (flops per byte of memory read). Thus, it is compute-bound.
During the **Decoding** phase, we generate one token at a time. The operations are vector-matrix multiplications (GEMV). We must load the entire model weight tensor from global memory (VRAM) to local registers for every single token generated. Because we do very few calculations per weight loaded, the processor is idle waiting for VRAM memory transfers, making it memory-bandwidth bound.

### Q2: What is the computational complexity of the attention computation during autoregressive generation *without* a KV Cache, and why?
Without a KV cache, for a prompt of length $P$ generating $M$ new tokens, at step $i \in [1, M]$, the model must calculate the attention representations of all $P + i - 1$ tokens from scratch.
For a sequence of length $N$, the self-attention computation scales quadratically $O(N^2)$ due to the pairwise query-key dot products. Accumulating this over $M$ decoding steps results in a total complexity of $O((P+M)^3)$ calculations instead of $O(M \cdot (P+M)^2)$ when caching past keys and values.

### Q3: Explain how Top-P (Nucleus) sampling works and why it is preferred over Top-K sampling in open-ended generation.
Top-K sampling restricts generation to the top $K$ tokens. However, the probability distribution changes shape depending on the context: in some contexts, the distribution is flat (many valid options), and in others, it is highly peaked (only one or two options are logical). 
Top-P sampling dynamically adjusts the number of candidate tokens. It selects the subset of highest probability tokens whose cumulative sum exceeds $P$ (e.g., $P=0.90$). If the distribution is peaked, the subset is small (avoiding gibberish); if the distribution is flat, the subset is large (maintaining creativity).
